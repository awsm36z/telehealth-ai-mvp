## 3. DELIVERABLES & ACCEPTANCE CRITERIA

### 3.1 Weekly Work Plan
By **Sunday 8:00 PM** of each week:
- Parties will agree on weekly deliverable(s) via email or project management tool
- Deliverable must be specific, measurable, achievable (SMART criteria)
- Estimated hours and completion deadline specified
- Acceptance criteria clearly defined

**Example Format:**
```
Week of [Date]:
- Deliverable: "Complete user authentication API endpoints"
- Acceptance Criteria:
  1. Registration endpoint functional (POST /api/auth/register)
  2. Login endpoint functional (POST /api/auth/login)
  3. JWT token generation working
  4. Input validation implemented
  5. Unit tests written (>80% coverage)
  6. Code committed to GitHub
  7. Postman collection provided
- Estimated Hours: 15 hours
- Due: Friday 6:00 PM
```

### 3.2 Deliverable Submission
- Developer submits deliverable by agreed deadline
- Submission includes: code, documentation, demo video (if applicable), time log
- Submission via GitHub pull request or agreed method

### 3.3 Acceptance Process
- Client reviews within **3 business days**
- Client provides one of three responses:
  1. **ACCEPTED** - Payment processed
  2. **ACCEPTED WITH MINOR REVISIONS** - Payment processed, revisions noted for next week
  3. **REJECTED** - Specific deficiencies listed, no payment until corrected

- If rejected: Developer has **2 business days** to remedy deficiencies
- If remedied: Payment processed for partial hours (pro-rated)
- If not remedied: No payment for that week, potential termination

### 3.4 Objective Quality Standards
Work must meet professional standards including:
- Code is functional and bug-free (no critical errors)
- Code follows agreed-upon style guide and best practices
- Code is properly commented and documented
- Git commits are logical with clear messages
- No plagiarized or copy-pasted code without attribution

### 1. **Compensation Rate ($25/hour)**
**Assessment:** Below market rate for developers

**Risks:**
- May only attract junior/inexperienced developers
- Higher turnover risk
- Quality concerns

**Recommendations:**
- **Option A:** Keep $25/hr but add 0.5-1% equity vested over 12 months

### 2. **Independent Contractor vs. Employee Classification**
**Risk:** IRS may reclassify as employee if:
- You control when/where/how they work (in-person requirement increases risk)
- They work exclusively for you
- Relationship is indefinite

**Mitigation:**
- ‚úÖ Contract clearly states independent contractor

- ‚ö†Ô∏è In-person requirement increases risk
- ‚úÖ Project-based with clear end date helps

**Recommendation:**

- Allow Developer to work for others
- Consider relaxing in-person requirement to "hybrid" (3 days in-person, 2 remote)

### 3. **Non-Compete Clause**
**Risk:** Non-competes for contractors are difficult to enforce in many states (CA, CO, etc.)

**Recommendation:**
- Keep it narrow (6 months, specific to telehealth AI, geographic limit)
- Focus more on non-solicitation and confidentiality (more enforceable)

### 4. **In-Person Requirement**
**Risk:** Significantly limits candidate pool, may deter good developers

**Recommendation:**
- Consider "hybrid" model: 3 days in-person, 2 days remote
- Or: "In-person for first 4 weeks, then remote with weekly in-person meetings"
- This increases candidate pool while maintaining collaboration

### 5. **Weekly Payment**
**Pro:** Good for cash flow management
**Con:** Administrative burden (weekly invoices, approvals, payments)

**Recommendation:**
- Consider bi-weekly payment after first 4 weeks (trial period weekly)
- Reduces admin overhead
- Still provides frequent milestones
